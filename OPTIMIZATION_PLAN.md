# ScreenshotEditor 优化计划

## 目标

把当前“可演示的截图编辑原型”推进到“功能闭环、行为一致、可测试、可维护的 macOS 应用”。

## 当前状态摘要

- 主 App target 可以构建成功
- 目录分层基本清楚，但 `AppState` 过大，承担了过多职责
- 功能面比 README/旧 TODO 更丰富，但文档明显落后于真实代码
- 存在多个关键闭环缺失：标注导出、批量导出交互、持久化恢复、测试 target

## 优化原则

1. 先修用户可见错误，再修内部结构
2. 先保证预览与导出一致，再继续叠加功能
3. 每修一个 bug，补一个回归测试
4. 文档必须反映真实工程状态，避免“纸面完成”

## 分阶段计划

### 阶段 0：建立真实基线

目标：确认“项目当前真实状态”，为后续优化提供可靠起点。

- [ ] 检查 Xcode target、scheme、test target 的真实配置
- [ ] 核对 README、TESTING、TODO、AGENTS 里的描述与工程现状
- [ ] 清点未接线功能、遗留文件、重复实现、占位逻辑
- [ ] 形成一次 baseline 记录：可构建性、测试状态、主要风险

验收标准：

- [ ] 文档不再声明不存在的测试或已完成但未闭环的功能
- [ ] 遗留文件和真实生效文件的边界清楚

### 阶段 1：修复 P0 用户问题

目标：先解决最影响可用性和可信度的问题。

#### 1. 标注导出闭环

问题：

- 画布预览会显示标注
- 导出流程没有渲染 annotations
- 用户会认为导出“丢内容”

任务：

- [ ] 为导出链路设计 annotation 渲染输入模型
- [ ] 将文字、箭头、矩形、高亮、模糊、马赛克等标注渲染进导出图像
- [ ] 对暂未真正支持的标注类型做降级策略或临时禁用
- [ ] 增加“预览与导出一致性”测试样例

验收标准：

- [ ] 有标注的截图导出后可见同等结果
- [ ] 至少覆盖文字、箭头、矩形、高亮四类标注

#### 2. 全局快捷键修复

问题：

- `register()` 会覆盖前一次注册
- 实际判断逻辑硬编码成 `command + shift`
- F3、Shift+F3、I 等注册逻辑不可信

任务：

- [ ] 将单一 `handler` 改为支持多个快捷键映射
- [ ] 按传入 modifiers 做精确匹配
- [ ] 为权限不足时提供明确引导
- [ ] 为截图、Pin、关闭 Pin、取色做手工回归验证

验收标准：

- [ ] `Cmd+Shift+K`、`F3`、`Shift+F3`、`I` 可同时工作
- [ ] 权限不足时行为清晰、不会静默失败

#### 3. 批量导出交互闭环

问题：

- 有多选模式 UI
- 有 `selectedScreenshotIds` 和 `exportBatch`
- 但列表没有真正维护多选集合

任务：

- [ ] 明确多选交互模式
- [ ] 将列表选择与 `selectedScreenshotIds` 绑定
- [ ] 提供批量导出入口、状态反馈、失败统计
- [ ] 增加空选、部分失败、全部成功三类验证

验收标准：

- [ ] 用户可以真实多选并触发批量导出
- [ ] 导出结果和失败提示清晰

#### 4. 清理覆盖层重复实现

问题：

- 存在 [Views/CaptureOverlayView.swift](/Users/eba/Desktop/ScreenshotEditor/ScreenshotEditor/Views/CaptureOverlayView.swift:1) 和 [Utilities/CaptureOverlayWindow.swift](/Users/eba/Desktop/ScreenshotEditor/ScreenshotEditor/Utilities/CaptureOverlayWindow.swift:1) 两套实现
- 当前 target 只编译其中一套，容易误导维护者

任务：

- [ ] 明确保留哪一套实现
- [ ] 删除或归档重复文件
- [ ] 统一调用入口和命名
- [ ] 补充注释说明覆盖层生命周期

验收标准：

- [ ] 项目里只有一套有效的截图覆盖层实现
- [ ] 后续维护者不会误改未生效代码

### 阶段 2：补齐测试与质量护栏

目标：把“能跑”推进到“能回归验证”。

#### 1. 建立真实 XCTest target

问题：

- 仓库有 `ScreenshotEditorTests/README.md`
- 但当前项目没有 test target

任务：

- [ ] 新建 `ScreenshotEditorTests` 单元测试 target
- [ ] 根据需要补 `ScreenshotEditorUITests`
- [ ] 更新 `TESTING.md` 中的真实命令和路径

验收标准：

- [ ] `xcodebuild test -project ScreenshotEditor.xcodeproj -scheme ScreenshotEditor -destination 'platform=macOS'` 可运行

#### 2. 首批回归测试清单

任务：

- [ ] `ImageExporterTests`：PNG/JPEG/WebP fallback、背景类型、圆角、阴影、边框、设备边框
- [ ] `AppStateTests`：初始状态、导入、删除、选中逻辑、错误处理
- [ ] `AnnotationExportTests`：标注导出一致性
- [ ] `PersistenceTests`：保存与恢复元数据
- [ ] `HotKeyMappingTests`：快捷键注册和匹配逻辑

验收标准：

- [ ] 新增逻辑都有对应测试
- [ ] 修 bug 必须带回归测试

### 阶段 3：修复状态管理与持久化

目标：降低状态混乱和“看似保存、实际没恢复”的问题。

#### 1. 持久化真正可用

问题：

- 当前只保存 metadata 和 PNG
- 加载时并没有真正恢复截图列表和设置

任务：

- [ ] 设计可编码的截图元数据模型
- [ ] 启动时恢复截图列表、设置、选中状态
- [ ] 处理丢失文件、损坏文件、版本升级
- [ ] 给“iCloud”命名降级，避免误导

验收标准：

- [ ] 重启应用后，历史和设置可恢复
- [ ] 遇到损坏数据时不会崩溃

#### 2. 拆分 `AppState`

问题：

- `AppState` 同时处理 UI、文件、截图、导出、快捷键、Pin、持久化
- 后续新增功能会继续放大耦合

建议拆分：

- [ ] `ScreenshotStore`：截图列表、选中状态、历史
- [ ] `EditorSettings`：背景、圆角、阴影、设备边框、导出偏好
- [ ] `CaptureCoordinator`：截图权限、覆盖层、区域截图
- [ ] `ExportCoordinator`：单张/批量导出、保存面板、剪贴板
- [ ] `AnnotationStore`：标注集合、工具状态

验收标准：

- [ ] `AppState` 只保留顶层组合职责，代码量显著下降
- [ ] 纯逻辑从 UI/系统 API 中分离，便于测试

### 阶段 4：统一渲染与交互体验

目标：让 UI 行为与用户预期一致。

#### 1. 统一预览与导出渲染链

问题：

- 预览使用 SwiftUI 组合
- 导出使用 Core Image/Core Graphics
- 两套实现天然容易漂移

任务：

- [ ] 建立共享渲染模型
- [ ] 明确各效果顺序
- [ ] 对阴影、边框、设备框、背景缩放做像素级校验
- [ ] 为差异较大的效果建立对照截图

验收标准：

- [ ] 同一组参数下，预览与导出视觉偏差足够小

#### 2. 文案与语言统一

问题：

- 当前中英混用明显
- 历史、标注、欢迎页、侧栏命名风格不一致

任务：

- [ ] 决定首发语言策略：中文优先或英文优先
- [ ] 统一按钮、标题、帮助文案、错误提示
- [ ] 统一“截图”“导入”“历史”“标注”等术语

验收标准：

- [ ] 主界面不再出现无规则中英混排

#### 3. 空状态与反馈优化

任务：

- [ ] 导入空状态更明确
- [ ] 批量导出过程提供进度和总结
- [ ] 截图权限失败、保存失败、导出失败提供可操作提示
- [ ] 操作成功时用更友好的 toast 或状态反馈替代 `print`

### 阶段 5：把“高级功能”从壳变成能力

目标：决定哪些高级功能继续做，哪些暂时收敛。

#### 1. Vision 能力产品化

现状：

- `VisionService` 功能不少
- 但没有真实业务入口

任务：

- [ ] 决定首批场景：OCR 文本框建议、高亮敏感信息、二维码识别、自动框选区域
- [ ] 设计入口按钮、结果展示、确认交互
- [ ] 为不可用系统版本设计降级

验收标准：

- [ ] 至少 1 个 Vision 能力进入可操作主流程

#### 2. Pin 窗口能力收敛

任务：

- [ ] 验证拖拽、缩放、旋转、透明度等是否都稳定
- [ ] 决定哪些手势保留，哪些简化
- [ ] 处理 workspace 保存/恢复与 group 逻辑的一致性

验收标准：

- [ ] Pin 功能不是“代码很多但体验不稳”

## 建议执行顺序

1. 标注导出闭环
2. 快捷键修复
3. 批量导出闭环
4. 清理重复实现
5. 建立测试 target
6. 补首批回归测试
7. 修复持久化恢复
8. 拆分 `AppState`
9. 统一渲染链
10. 再决定 Vision / Pin 的深化路线

## 风险清单

- [ ] 预览与导出双实现长期分叉
- [ ] 全局快捷键与系统权限耦合，调试成本高
- [ ] AppKit 面板和 SwiftUI 状态混用，容易出现线程和生命周期问题
- [ ] 没有测试护栏时，修一个功能容易破另一个功能

## 完成定义

满足以下条件时，可认为项目从“原型期”进入“可持续开发期”：

- [ ] build 和 test 都可稳定运行
- [ ] 预览与导出一致
- [ ] 标注、批量导出、持久化都形成闭环
- [ ] 文档和工程状态一致
- [ ] 至少有一层可靠的自动化回归测试

## 维护方式

建议每完成一个阶段，就同步更新：

- `TODO.md`：勾选已完成项，调整近期优先级
- `CHANGELOG.md`：记录用户可感知变化
- `TESTING.md`：补充新增测试与回归命令
